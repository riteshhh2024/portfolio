# GitGrow Agent - Developer Productivity Assistant

## Overview

GitGrow Agent is a developer productivity assistant designed to work inside a Git repository. It scans real project changes, understands what was built, helps update project documentation, generates clean commit messages, and supports a safer commit-and-push workflow.

<GitGrowPipeline />

The goal is not to create fake GitHub activity. The goal is to help developers build a professional and consistent GitHub presence through **real work**, **better documentation**, and **cleaner project history**.

## Problem Statement

Developers often write code but forget to maintain the supporting project information around that code. This creates several problems:

- **Stale documentation**: README files, changelogs, API docs, and setup instructions go out of date after code changes.
- **Unclear commit history**: Repositories are filled with messages like `update`, `fix`, `final`, or `changes`.
- **Invisible progress**: Students, freelancers, and job seekers need visible proof of consistent and meaningful project activity.
- **Lost context**: Developers lose context when returning to projects after days or weeks.
- **Documentation drift**: Teams suffer when documentation diverges from the actual codebase.

## Solution

GitGrow follows a simple, transparent workflow:

```
Code changes → scan diff → classify work → update docs → generate commit → preview → approve → commit/push → progress log
```

The agent supports the developer without silently taking control. By default, GitGrow **previews changes and asks for approval** before committing or pushing.

## Key Features

### Smart Commit
Generates professional **Conventional Commit** messages from the actual git diff. No more vague commit messages — every commit tells a clear story about what changed and why.

### DocSync
Automatically updates **README**, **CHANGELOG**, **API docs**, **setup docs**, and **.env.example** when major changes are detected. Documentation stays in sync with the code.

### Project Scanner
Reads **git status**, **diff**, **logs**, **changed files**, **dependencies**, **routes**, and **environment variables** to build a complete picture of the project state.

### Safety Checker
Runs **tests**, **lint commands**, **build commands**, and **secret checks** before committing. Ensures code quality gates are met before any changes are pushed.

### DevLog
Creates daily or weekly **progress logs** that summarize real development activity. Helps developers track what they built and when.

### GitHub Auth
Uses **GitHub CLI** for secure authentication without storing raw tokens. Security-first approach to credential management.

### Preview System
Shows all proposed docs, commit messages, and files before any action is taken. The developer always has the final say.

## Technical Implementation

### Agent Workflow Pipeline

The core pipeline processes changes through a structured series of stages:

1. **Scan Phase** — Read git status, diff, and project metadata
2. **Classification Phase** — Categorize changes by type (feature, fix, refactor, docs, etc.)
3. **Documentation Phase** — Detect which docs need updates based on change classification
4. **Generation Phase** — Create commit messages and documentation patches
5. **Preview Phase** — Present all proposed changes for developer review
6. **Execution Phase** — Apply approved changes, commit, and optionally push

### Diff Classification Engine

The scanner reads the raw git diff and classifies each change:

- **New files** — Feature additions, test files, configuration
- **Modified files** — Bug fixes, refactors, enhancements
- **Deleted files** — Cleanup, deprecation
- **Dependency changes** — Package updates, new dependencies
- **Environment changes** — New env variables, config updates

### Safe Documentation Updates

DocSync follows strict boundaries to avoid destructive updates:

- Never rewrites the entire README
- Preserves existing developer-written sections
- Only appends or updates relevant sections (features, API, setup, changelog)
- Uses a diff-based approach to minimize changes
- Timestamps are not updated unless meaningful content changes


## User Setup Flow

```bash
pip install gitgrow

cd my-project
gitgrow init
gitgrow auth github
gitgrow scan
gitgrow commit
```

1. Install GitGrow using pip
2. Open a local GitHub project repository
3. Run `gitgrow init` to create project configuration
4. Run `gitgrow auth github` to connect securely through GitHub CLI
5. Run `gitgrow scan` to inspect project changes
6. Run `gitgrow commit` to update docs, generate a commit, preview, and commit after approval

## What GitGrow Should Not Do

GitGrow is designed to be safe and honest by default:

- ❌ Create meaningless empty commits
- ❌ Update README only to change a timestamp
- ❌ Push code without user approval by default
- ❌ Store GitHub tokens in plain text
- ❌ Rewrite the entire README without safe boundaries
- ❌ Replace developer judgment
